Downloading Sequencing Data§

Most journals require that raw sequencing data be deposited to public databases.

  1. Researchers in the US usually deposit data in the Short Read Achive (SRA) hosted by the NCBI.

  2. European groups, usually deposit data in Array Express hosted by EMBL.

  3. Project Specific downloads:

    The Cancer Genome Atlas (TCGA)

    ENCODE

    1000 Genomes Project

Getting data from the SRA§

  1. Find specific experiments in the Gene Expression Omnibus from GEO Datasets.
  2. Downloading "processed" experiment data. BED, Bedgraph, count tables, bigWig, etc.
  3. Extract the SRA accession numbers and download raw data from the SRA GUI
  4. Use fastq-dump on command line.

fastq-dump§

The SRAToolKit has command-line utilities to allow for programmatic access to sequencing data. The fastq-dump command is useful for directly downloading fastq data, provided a SRA accession number.

~/path_to_sratoolkit/bin/fastq-dump -h
# -X download only a specific number of records

fastq-dump -X 10 --gzip SRR390728

Download in a loop§

Use SRA website to get a list of SRA accesion number for a project. Then download all of the data as fastq data.

files=$(cat sranumbers.txt)

for x in $files
  do
  echo "downloading "$x
  fastq-dump -X 10 --gzip $x
  done

Download in parallel§

GNU parallel is a powerful unix tool for parallelizing jobs if you don't have access to a compute cluster.

#same output as previous for loop
# -j indicates number of jobs to run at once

cat sranumbers.txt \
  | parallel -j 4 `fastq-dump -X 10 {}`

Getting help§

Sometimes you will get errors that you do not understand.

  1. First google it. Quote the error message and include the package name or language. If there are variables or files included in the message exclude them from the search.
  2. Go to the source code repository and search for your problem (i.e. github issues, google groups/mailing list)
  3. Before you ask others for help, make sure the question hasn't been answered before.

There will be bugs§

  1. Bioinformatics relies on software built by people and people make mistakes. There is a lot of bad code out there.

  2. There will be bugs and most tools won't do everything you wish they would.

  3. Solutions:

    get developers to fix the bug

    fix the bug yourself and contribute to the project

    use another tool

    write your own tool

Reproducible Example (reprex)§

  1. Writing a reproducible example is a key skill to getting any help from the internet.
  2. Keep it simple and distill your problem down to a small set of code the reproduces your error.
  3. The error should be recaptitulated by copying and running the code as is. Do not depend on external data if possible
  4. Often by trying to simplify and reproduce your error you will actually solve your problem.

What makes a good reprex?§

  1. load necessary package at top
  2. use built-in datasets if possible, otherwise use dput(), or tibble::frame_data()
  3. Use comments to explain steps
  4. Include relevant error message
  5. Potentially include the output of sessionInfo()

The reprex package§

The reprex package makes it easy to produce well-formatted R reprexes for posting to github or stackoverflow.

install.packages("reprex")
library(reprex)

copy the following code into your clipboard

a <- c("1", "1.2")
mean(a)

Now in the console type:

reprex()
